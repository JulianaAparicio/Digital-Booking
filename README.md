**<span dir="">Resumen del proyecto:</span>**

<span dir="">Los integrantes del equipo culminan con esta instancia integradora de la carrera Certified Tech Developer, diseñada por Digital House junto con Mercado Libre y Globant. A lo largo de esta cursada se han adquirido amplios y diversos conocimientos de las distintas áreas del desarrollo de software web. Dentro de las habilidades técnicas se han recorrido temáticas de informática, base de datos, testing, infraestructura, lenguajes de programación y así mismo se han adquirido habilidades blandas fundamentales hoy en día para desempeñarse adecuadamente en equipos de trabajo y dentro del actual mercado laboral. </span>

<span dir="">En conjunto estos conocimientos han fundado las bases para facilitar, allanar el camino en cuanto al trabajo en equipo y culminar este proceso educativo llevando a cabo el proyecto integrador.  </span>

<span dir="">Digital Booking es el producto final que se trabajó desde principio a fin en esta última etapa evaluativa. El Proyecto apunta a una plataforma digital la cual está orientada al alquiler de diferentes categorías alojamientos, en la cual se pueden publicar/crear alojamientos desde el rol del usuario administrador y reservar alojamientos desde el rol de usuario de la aplicación.</span>

**<span dir="">Desde el rol de usuario de la aplicación:</span>** 

* <span dir="">Se puede dar de alta/crear un nuevo usuario con los datos necesarios. </span>
* <span dir="">Se puede iniciar sesión con credenciales válidas, si el usuario ya está creado previamente.</span>
* <span dir="">Se puede filtrar un alojamiento según un listado de ciudades y un rango de fechas especificadas en la búsqueda. </span>
* <span dir="">En el home se puede visualizar un preview de la ubicación de cualquier alojamiento mostrado en el listado de Recomendaciones, haciendo click en el botón “mostrar mapa”, funcionalidad disponible dentro de cada card de producto en particular. </span>
* <span dir="">Dentro de cada producto (alojamiento) en particular se encuentran disponibles: </span>
  * <span dir="">Nombre y ubicación del alojamiento</span>
  * <span dir="">Galería de imágenes del alojamiento </span>
  * <span dir="">Descripción del alojamiento</span>
  * <span dir="">Amenities que ofrece el alojamiento</span>
  * <span dir="">Un calendario que muestra la disponibilidad del producto</span>
  * <span dir="">Un mapa en detalle donde se puede observar la ubicación del alojamiento</span>
  * <span dir="">Normas y políticas del alojamiento en particular</span>
* <span dir="">Una vez filtrado el producto por fechas y ciudad se puede proceder a la reserva de este si el usuario está logueado previamente en la aplicación. </span>
* <span dir="">Se pueden consultar las reservas efectuadas por el usuario, accediendo al perfil del usuario logueado consultando en “Mis Reservas”.</span>
* <span dir="">Se pueden fijar productos favoritos en el home, los mismos pueden ser consultados accediendo al perfil del usuario logueado consultando en “Mis Favoritos”. </span>
* <span dir="">La aplicación posee la funcionalidad de “Crear Cuenta” y de “Inicio de Sesión”. Se deberán autenticar previamente las credenciales siempre que se quiera realizar una reserva, consultar las reservas o consultar favoritos del usuario. Una vez creado el usuario con éxito se recibirá un correo electrónico confirmando el registro exitoso.</span>

<span dir="">Una vez que el usuario ya está registrado, el mismo tiene acceso a la interfaz de reserva del producto que desee según sus criterios de búsqueda. Dentro de esta interfaz se verifican los datos del usuario, fechas de entrada y salida, horario de llegada, estado de vacunación sanitario, y si se quisiera se podrán modificar las fechas de reservación dentro del calendario. Una vez chequeado todo, el usuario podrá realizar una reserva exitosa, recibirá un correo de confirmación de la misma y se lo redirigirá a la reserva realizada. </span>

**<span dir="">Desde el rol de usuario administrador de la aplicación:</span>**

<span dir="">Con el rol de administrador se puede acceder a la interfaz de “Crear Producto” dentro de la cual se podrán crear nuevos alojamientos. El usuario debe iniciar sesión con credenciales válidas para el rol de administrador y en la interfaz visualizará campos para la creación del producto:</span>

* <span dir="">Nombre del alojamiento</span>
* <span dir="">Categoría</span>
* <span dir="">Dirección</span>
* <span dir="">Ciudad</span>
* <span dir="">Descripción</span>
* <span dir="">Amenities</span>
* <span dir="">Políticas </span>
* <span dir="">Imágenes: son cargadas a través de URL </span>

<span dir="">Otra de las funcionalidades del rol de administrador es la de “Crear Usuarios”. Aquí el administrador podrá crear tanto usuarios comunes que consumen la aplicación como otros usuarios con permiso de administrador. En esta interfaz visualizará los campos para la creación del usuario:</span>

* <span dir="">Tipo de usuario</span>
* <span dir="">Nombre</span>
* <span dir="">Apellido</span>
* <span dir="">Correo electrónico</span>
* <span dir="">Contraseña</span>

Para lograr el objetivo se abarcan todas las áreas de estudio vistas en la carrera de Certified Tech Developer. </span>

**<span dir="">Asignación de roles:</span>** 

<span dir="">Para la asignación de roles se tuvieron en cuenta las fortalezas expuestas por cada miembro del equipo y las ganas de incursionar en habilidades no desarrolladas a lo largo de la cursada de la carrera. **<span dir="">Metodologías de Trabajo:</span>** 

<span dir="">Metodología Scrum:</span>

<span dir="">A lo largo del Proyecto Integrador se trabajó con la metodología Scrum que se dividió en 4 Sprints, de 4 semanas de duración cada uno. </span>

<span dir="">En el comienzo de cada Sprint se llevó a cabo la lectura de los requerimientos del proyecto y las historias de usuario. Luego en la planificación el equipo de trabajo analizó en profundidad cada uno de los requerimientos, los organizó en un tablero de trabajo y se repartieron las tareas entre los miembros y se estimaron los tiempos de cada una de las tareas asignadas. </span>

<span dir="">Durante el desarrollo del sprint, todos los días se realizó la correspondiente reunión de daily en equipo junto con el Scrum Máster, en la cual se discutían diversos tópicos:</span>

* <span dir="">Avances en las tareas asignadas </span>
* <span dir="">Obstáculos o bloqueos significativos </span>
* <span dir="">Detalle de las tareas a desarrollar durante el resto del día</span>

<span dir="">El segundo lunes de cada Sprint se llevó a cabo la weekly en la cual estaban presentes tanto el Scrum Máster como el Product Owner. En la ceremonia se hablaba del desarrollo de las tareas llevadas a cabo hasta el momento, el avance del equipo, bloqueos no resueltos y plan de trabajo para la semana entrante. </span>

<span dir="">Al finalizar cada sprint se cerraba con una ceremonia de retrospectiva en la cual cada integrante del equipo discutía los que consideraban puntos débiles del sprint pasado e ideas de mejoras para el siguiente. </span>

**<span dir="">Bitácora del Proyecto:</span>** 

* <span dir="">Sprint 1:</span>

<span dir="">El mayor desafío del primer sprint fue conocernos como equipo ya que nunca habíamos trabajado juntos anteriormente y ninguno estaba muy familiarizado con el uso de la herramienta de GitLab. Decidimos utilizar Trello para facilitar la asignación de tareas del sprint. Lo primordial fue lograr los templates generales de la UI, las variables de estilos y la creación de la base de datos. </span>

* <span dir="">Sprint 2:</span>

<span dir="">Durante el segundo sprint ya nos habíamos logrado organizar de mejor manera, pudimos optimizar el tiempo y flujo de trabajo. Logramos llegar a cumplir con las funcionalidades extras y eso como equipo nos dio la confianza de que íbamos por el camino correcto. </span>

* <span dir="">Sprint 3:</span>

<span dir="">En el tercer sprint nos enfocamos en corregir algunos detalles de la UI, se corrigió la calidad de algunas imágenes de la base de datos. Se logró cumplir con todos los requerimientos. Pudimos hacer el deploy de manera conjunta. </span>

* <span dir="">Sprint 4:</span>

<span dir="">Llegamos en una buena posición ya que no nos quedaban tareas pendientes de los sprints anteriores. Pudimos concentrarnos en los requerimientos nuevos, en pulir algunas cuestiones de la UI y trabajamos en la presentación de manera conjunta al igual que en la entrega de los informes finales. </span>

<span dir="">Una de nuestras mayores fortalezas como equipo fue la constante comunicación, los miembros del equipo mantuvimos siempre una comunicación fluida, con respeto y armónica y eso se vio reflejado en el flujo de trabajo continuo. Supimos apoyarnos mutuamente y ayudarnos cuando fue necesario, aprendimos todos algo nuevo en conjunto siempre. </span>

<span dir="">El mayor obstáculo que enfrentamos fue el de la organización del tiempo y división de tareas. Al no haber trabajado nunca juntos, nos tomó un poco de tiempo conocernos y ver cómo funcionamos a nivel equipo. Afortunadamente logramos establecer una forma de trabajo en la que todos pudimos colaborar. </span>

**<span dir="">Ambiente de desarrollo</span>**

<span dir="">Instrucciones de cómo un nuevo integrante debería armarse su ambiente de desarrollo:</span>

* <span dir="">Front-End</span>
  * <span dir="">Dependencias: Node.js, npm, git, vite</span>
  * <span dir="">Para utilizar la aplicación, en una carpeta y en la terminal, clonar el repositorio.</span>
    * <span dir="">$ git clone url</span>
  * <span dir="">En la carpeta de frontend instalar los paquetes de node.</span>
    * <span dir="">$ npm install</span>
  * <span dir="">Para iniciar el proyecto.</span>
    * <span dir="">$ npm run dev</span>

<span dir="">Para el desarrollo FrontEnd de este proyecto se utilizó Javascript con React como librería principal, la estructura de SPA se mantuvo utilizando la dependencia React-router-dom como enrutador, y el estilizado fue hecho utilizando SASS(Scss) por medio de una dependencia del mismo nombre que permite importar ese tipo de archivos directamente en JS sin necesidad de compilarlos a CSS vanilla primero. Y hablando del aspecto estético, las animaciones, transiciones y demás efectos de la aplicación fueron realizados utilizando una library llamada GSAP (Greensock Animations Platform).</span>

<span dir="">Para las conexiones a la API se utilizó Axios para alivianar la sintaxis, sobre todo a la hora de hacer Posts.</span>

<span dir="">Para crear la estructura del proyecto de lado del front se utilizó Vite, la cual trae incluidas y configuradas out of the box NPM como gestor de dependencias y Es-build como bundler, pre procesador y compilador. Fue elegido Vite por sobre CRA por su notable diferencia de tiempo a la hora de compilar tanto los cambios en tiempo real, como al momento de hacer un build para producción, esto gracias al bundler que se utiliza, el cual en CRA es Webpack, y por su menor cantidad de dependencias incluidas por defecto.</span>

<span dir="">Los testings unitarios y de componentes se hicieron utilizando Vitest la cual es la library oficial de Vite para realizar tests cuando se utiliza su inicializador de proyectos.</span>

* <span dir="">Backend</span>
  * <span dir="">Para correr el back se debe abrir el mismo en su IDE de preferencia por ejemplo Intellij.</span>
  * <span dir="">Seleccionar la carpeta BackEnd</span>
  * <span dir="">Ejecutar el proyecto</span>

<span dir="">En relación con el Backend se utilizó el lenguaje Java con Springboot como framework principal del proyecto, utilizando el patrón MVC (Modelo-Vista-Controlador) para hacer la separación de la lógica de la vista y de la aplicación.</span>

<span dir="">A su vez, la gestión de todo el proyecto y principalmente la de las dependencias que se utilizaron fue realizada íntegramente con Maven, así como también la ejecución automática de los tests.</span>

<span dir="">En cuanto a la API se utilizó Swagger para desarrollar la documentación de todos los controller del proyecto.</span>

<span dir="">Para realizar los tests unitarios nos enfocamos principalmente en la capa del service ya que es donde se aloja la mayor parte de la lógica del negocio. Se utilizó la librería Junit 5 en conjunto con Mockito para evitar la persistencia de datos generados por la ejecución de los tests.</span>

<span dir="">Buenas prácticas / convenciones / acuerdos a la hora de desarrollar.</span>

* <span dir="">Buenas prácticas respecto a los commits, merges y el uso de branches.</span>
  * <span dir="">Se acordó siempre trabajar en ramas individuales creadas a partir de la de desarrollo (main).</span>
* <span dir="">Nunca trabajar sobre la rama principal, main.</span>
* <span dir="">Usar etiquetas lo más descriptivas en los commits</span>
* <span dir="">Antes de un merge siempre avisar a los compañeros de equipo para que estemos todos al tanto</span>
* <span dir="">Realizar pull al comienzo de cada jornada y cada vez que se informe de un cambio importante realizado por las áreas del proyecto.</span>
* <span dir="">Se realiza apoyo a las áreas que requieran atención de acuerdo a los problemas y bloqueos que presenten.</span>
